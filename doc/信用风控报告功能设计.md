# 信用风控报告 · 功能设计

> 适用产品：同盾风格「贷前风控 SaaS」— 申贷审核主线 · 信用风控（联防联控）模块
> 文档定位：信用风控环节的「展示 + 判定 + 操作」载体设计，供信审专员、风控策略岗、风控管理员使用。
> 关联文档：`doc/申贷审核功能详细设计.md`、`doc/信息核验报告功能设计.md`、用户提供的《金融风控贷前审核-信用风控（联防联控）完整设计文档》

## 一、功能定位与边界

信用风控报告是**申贷审核流程中「信用风控（联防联控）」环节的结果载体**。它在「信息核验」通过后执行，把"用户入参 + 内部存量 + 人行征信 + 同盾跨行业联防 + 失信/运营商履约 → 指标计算 → 交叉融合 → 决策分流"的全过程，沉淀为一份可被查看、可被操作、可被监管留痕的结构化报告。

### 1.1 与「信息核验报告」的关系
| 维度 | 信息核验报告 | 信用风控报告 |
|---|---|---|
| 关注点 | 信息**真假**（证件/实名/设备） | 信用**历史、全行业借贷行为、跨机构风险** |
| 数据源 | 公安/运营商/银联/设备/联防 | 人行征信/内部信用库/同盾联防/失信/运营商履约 |
| 输出分 | 核验置信分 | 信用综合分（同盾分） |
| 触发 | 进件提交即触发 | 信息核验完成后触发（可并行） |

### 1.2 报告要回答的 5 个问题
| 问题 | 对应报告区块 |
|---|---|
| 系统**聚合了哪些入参**？ | 3.1 用户提交与入参（原始） |
| 这些入参被**加工成哪些信用特征**？ | 3.2 结构化信用特征 |
| 每个数据源**单项**判定如何？ | 3.3 单项信用判断结果 |
| 风控引擎**是怎么跑的**？ | 3.4 信用风控过程 |
| 最终**信用结论**是什么？ | 3.5 信用结论（含交叉融合） |

### 1.3 报告上的两类操作
- **单条信用异常操作**（3.6）：信审员对**某一条联防异常**（如"失信被执行人""团伙关联"）做二次核验、标记、备注、电核。
- **整份报告全局操作**（3.7）：对整份报告做自动通过 / 人工放行 / 降额授信 / 拒绝 / 退回补件 / 存证 / 归档等。

### 1.4 输入 / 输出
- **输入**：进件基础入参 + 内部存量库 + 外部数据源回执 + 交叉融合引擎输出。
- **输出**：① 机器结构化报告（JSON，给决策引擎/天策调用）；② 人工可视化报告（本设计主体）；③ 监管存证包（征信/联防回执+操作日志）。

## 二、报告核心数据模型（字段总览）

```
Report
├─ report_id              报告唯一 ID
├─ apply_no               进件申请号
├─ applicant              申请人基础信息（脱敏）
├─ status                 报告状态（待审核/审核中/已通过/已拒绝/已退回/已归档）
├─ raw_inputs             ① 用户提交与入参（基础入参 + 内部存量 + 外部请求）
├─ structured            ② 结构化信用特征（多头/负债/失信/关联）
├─ single_results[]       ③ 单项信用判断结果（5 大数据源各自结果）
├─ credit_process         ④ 信用风控过程（并行任务、耗时、回执、熔断）
├─ credit_conclusion      ⑤ 信用结论（交叉融合 + 同盾分 + 风险等级 + 标签 + 分流）
├─ item_actions[]         ⑥ 单条信用异常操作记录
├─ report_actions[]       ⑦ 整份报告全局操作记录
└─ audit_trail            全链路操作日志（监管留痕）
```

> 设计原则：**原始入参、结构化特征、单项结果、风控过程、信用结论、单条操作、整体操作** 7 个区块一一对应产品诉求。

## 三、报告功能模块详解

### 3.1 用户提交与入参（原始）
**目的**：还原信用风控引擎的"输入基准"——系统自动聚合的三类数据，满足监管"数据来源可追溯"要求。

展示内容（三类分组）：
- **用户结构化基础入参**：姓名、身份证号、手机号、银行卡、设备指纹ID、申请IP、年龄、收入、工作、申请金额/期限。
- **内部自有存量数据**：本行历史申请/放款/逾期/催收、黑名单灰名单、公积金/社保/流水/房车资产、产品准入阈值配置。
- **外部第三方联防请求**：人行征信授权状态、同盾联防调用状态、失信查询、运营商履约调用。

交互：分组折叠展示；点击某条外部请求可展开调用状态/授权时间。

### 3.2 结构化信用特征
**目的**：展示系统把"原始三源"加工成的可比信用特征，作为后续打分的输入。

展示内容（特征表）：
| 特征类 | 字段 | 示例值 |
|---|---|---|
| 多头借贷 | 近7/30/90天申贷平台数、未结清笔数、总负债、申贷频次 | 近30天 5 家 / 总负债 15万 / 高 |
| 偿债能力 | 月负债、月收入、负债收入比 | 月负债 6800 / 收入 12000 / **56.7%** |
| 失信 | 失信被执行人、工商失信 | 无 / 无 |
| 关联关系 | 关联身份证/手机号/设备数、团伙风险等级 | 关联设备 5 / **G-014(高)** |

交互：异常特征高亮（如负债收入比>100% 红标）；可展开"特征→数据源"血缘。

### 3.3 单项信用判断结果
**目的**：呈现 5 大数据源**各自独立**的信用判定状态，便于定位"哪一类信用风险"。

5 大数据源卡片（每个含：状态徽标 / 输出字段 / 回执编码 / 耗时 / 风险标签）：
1. **人行征信**：征信概要、历史贷款、当前逾期、查询次数。
2. **内部信用库**：本行历史、逾期/催收、黑名单灰名单。
3. **同盾联防库**：多头借贷、跨机构逾期黑名单、团伙关联图谱、欺诈历史。
4. **失信黑名单**：法院失信被执行人、工商失信。
5. **运营商履约**：在网时长、还款履约、消费履约、虚拟号特征。

状态三态：`通过(绿)` / `存疑(黄)` / `失败(红)`。

### 3.4 信用风控过程
**目的**：具象化"并行执行 + 熔断降级"，让信审员/运维看到引擎真实执行轨迹，而非黑盒。

展示内容：
- **并行任务时间线**：1 次 `submit` 拆 5 条独立线程（人行征信1.2s / 内部库0.06s / 同盾0.8s / 失信0.4s / 运营商0.5s），各自下发/返回时间戳、耗时。
- **单渠道调用日志**：每条约 数据源 / 请求参数脱敏 / 返回码 / 原始回执ID / 状态。
- **熔断降级标记**：某渠道超时时显示"XX数据缺失（已熔断，不影响其他源）"。
- **聚合节点**：全部渠道返回后进入交叉融合计算的聚合时间戳。

### 3.5 信用结论（含交叉融合）
**目的**：报告的"结论页"——把单源结果融合为交叉冲突、信用分、标签与分流指令。

三部分：
1. **交叉融合清单 `cross_fusion`**：每条标注严重度（黄/红）与冲突来源。
   - 例🔴：失信被执行人 + 负债收入比131% → 极高风险（失信黑名单 VS 内部负债）
   - 例🔴：单一客户信息正常，但关联5个逾期账号(G-014) → 团伙欺诈（同盾联防 VS 内部/公安）
2. **信用综合分（同盾分）**：0–100，分数越高信用风险越高；由单源结果 + 交叉冲突数量 + 联防负面记录加权计算。
3. **风险等级 + 风险标签 + 决策分流指令**：
   - 低风险（分≤20）：多头≤3家、无失信、无团伙 → **自动通过**
   - 中风险（分 20–80）：多头4–8家、轻度逾期 → **人工复核 / 降额授信**
   - 高风险（分≥80）：失信被执行人 / 多头≥10家 / 团伙欺诈 → **直接拒绝**

### 3.6 操作人对单条信用异常的进一步操作
**目的**：信审员对**某一条联防异常**（如"团伙关联 G-014"）做人工干预，纠正误报、补充核验或确认欺诈。

单条操作类型（每个 single_result 卡片右上角"操作"菜单）：
| 操作 | 说明 | 落库 |
|---|---|---|
| 调取完整明细 | 查看多头机构清单、历史逾期时间、团伙关联图谱 | item_actions + audit_trail |
| 二次核验 | 重新调用该数据源（如二次同盾联防/人行征信） | 同上 |
| 电话电核 | 核实负债/多头真实用途，记录通话笔录 | 同上 |
| 风险标记 | 区分「合理周转多头」/「恶意撸贷」/「团伙欺诈」 | 同上 |
| 补充佐证 | 上传收入流水、结清证明，豁免部分风险 | 同上 |
| 添加备注 | 文字说明（必填理由） | 同上 |

约束：单条操作**不影响**其他源结果，也不自动改变整份报告分流；需结合 3.7 整体操作生效。

### 3.7 整份信用风控报告的全局操作
**目的**：对整份报告做出最终业务处置，是信用风控流程的出口。

整体操作类型：
| 操作 | 触发条件/适用 | 系统动作 |
|---|---|---|
| 自动通过 | 全绿、分≤20 | 进入授信定价 |
| 人工放行 | 中风险经核实无压力，推翻系统拒贷 | 正常授信，记录理由 |
| 降额授信 | 轻度多头/负债偏高 | 降低额度后通过 |
| 直接拒绝 | 红标/分≥80（失信/团伙/重度多头） | 记录欺诈台账，终止流程 |
| 退回补件 | 材料缺失可补 | 向用户下发补件通知 |
| 提交复审 | 高风险大额件 | 推送二级审核双人复核 |
| 风险入库 | 确认欺诈 | 加本地黑名单 + 回流联防样本库 |
| 导出/存证/归档 | 任意/终态前 | PDF导出；影像+回执+日志存证；留存≥5年 |

权限：信审专员可"转人工/退回/备注/电核"；风控审核员（二级）可"拒绝/通过/降额/放行"；风控策略岗可"配置阈值/黑白名单/样本回流"。

## 四、信用风控列表页设计（涉及的列表页修改）

> 当前 `src/console/specsCr.ts` 的 `cr:pre-credit`（信用风控列表页）仅有 6 列（进件号/申请人/近30天申贷平台数/涉及机构数/负债收入比/信用结论），且**未挂接报告详情**（无 `reportKey`/`viewNavigate`），是"断头列表"。
> 为支撑本报告载体，建议对列表页做如下改造（**设计建议，待确认后落地代码**）：

### 4.1 新增/调整列
| 列 | 字段 | 说明 |
|---|---|---|
| 进件号 | `id` | 增加 `reportKey: 'id'` 与 `viewNavigate: 'credit-report-detail'`，点击跳转报告 |
| 申请人 | `name` | 脱敏 |
| 同盾分/信用分 | `creditScore` | 新增分数列（0–100，红高分） |
| 近30天申贷平台数 | `plat` | 保留 |
| 涉及机构数 | `org` | 保留 |
| 负债收入比 | `debt` | 保留（百分比） |
| 信用结论 | `concl` | 正常/关注/高风险（徽标） |
| **报告状态** | `status` | **新增**：待审核/审核中/已通过/已拒绝/已退回/已归档 |
| **决策结果** | `decision` | **新增**：自动通过/人工复核/降额/拒绝/退回（与信用结论区分） |

### 4.2 状态筛选与跳转
- 顶部增加"报告状态"筛选（待审核/审核中/已通过/已拒绝/已退回）。
- 列表行点击"查看报告"→ 打开 `信用风控报告详情`（本设计 3.1–3.7 的渲染页）。
- 与现有"存疑进件"页联动：人工复核件可一键转入存疑进件工作台。

## 五、报告状态机与权限

```
            submit(信息核验通过)
              │
              ▼
         [待审核] ──自动全绿(分≤20)──▶ [已通过] ──▶ [已归档]
              │
              ├─中风险──▶ [审核中] ──信审判定──▶ [已通过(含降额)]/[已退回]/[已拒绝] ──▶ [已归档]
              │
              └─红标(失信/团伙/重度多头)──▶ [已拒绝] ──▶ [已归档]
```

- 任一终态（通过/拒绝/退回）前，必须完成**存证**。
- 所有状态变更写入 `audit_trail`（操作人、时间、前后值、理由）。

## 七、与同盾方案的映射（实现参考）

| 本报告区块 | 同盾对应能力 |
|---|---|
| 3.1 原始入参 | 机构 `submit` 进件报文（信息核验+信用联防同接口触发） |
| 3.2 结构化特征 | 同盾指标解析引擎（多头/负债/关联拆解） |
| 3.3 单项结果 | 单源 `data_source_list` 独立返回（人行/内部/联防/失信/运营商） |
| 3.4 信用风控过程 | 分布式多线程并行引擎（与信息核验同步执行） |
| 3.5 交叉融合 | 信用分（同盾分）+ 交叉冲突标签 |
| 3.6 单条操作 | 信审后台"疑点复核"单源操作 + 云图关联图谱展开 |
| 3.7 整体操作 | 天策决策引擎分流 + 名单管理 + 样本回流 |

> 同盾差异化核心：① 信息核验与信用联防**同一接口同步执行**（一体化并行）；② **跨机构多头联防**（自建系统无法获取他行借贷记录）；③ **团伙欺诈图谱**（GNN 穿透关联）；④ 风险**交叉联动**（证件虚假+多头+失信叠加）。本 SaaS 原型按同盾能力完整呈现上述差异。

- [ ] 原始入参：三类数据（基础/内部/外部）均可展示、可追溯

- [ ] 信用风控过程：并行时间线 + 熔断标记可见
- [ ] 交叉融合：冲突清单带严重度 + 同盾分 + 分流指令
- [ ] 单条操作：6 类操作可触发并写入 item_actions + audit_trail
- [ ] 整体操作：8 类操作可按权限触发并驱动状态机
- [ ] 列表页：新增 同盾分/报告状态/决策结果 列，点击跳转报告详情
- [ ] 多状态样例（见 6 章）可一键注入联调
