## 1. 决策结果 与 审核状态 的关系

### 1.1 核心原则

- **两个维度，职责不同，不能重叠。**
- **"转人工"是动作（路由），不是状态。** 它是从"审核中"跳到"待人工复核"的那个操作。
- **决策结果由最终的审核状态推导出来**，不独立存一个"转人工"结论。
- **准入 / 拒绝 一旦成立，就是终态，不可能再"转人工"。** 转人工只发生在还没出终局结论之前。

### 1.2 两个维度定义

| 维度 | 本质 | 取值 |
|---|---|---|
| **审核状态** | 流程走到哪一步（过程态） | 待审核 / 审核中 / 待人工复核 / 已通过 / 已拒绝 / 已关闭 |
| **决策结果** | 最终结论是什么（结论态） | 处理中 / 待人工判定 / 准入 / 拒绝 / 关闭 |

### 1.3 推导关系

```
审核状态           → 决策结果
待审核 / 审核中     → 处理中（未出结论）
待人工复核         → 待人工判定（等人拍板）
已通过            → 准入
已拒绝            → 拒绝
已关闭            → 关闭
```

## 2. 审核状态机

### 2.1 状态流转

```
       提交
待审核 ──→ 审核中
              │
    ┌─────────┼──────────────┐
    │自动准入   │自动拒绝       │[动作:转人工]
    ▼          ▼             ▼
  已通过     已拒绝       待人工复核
                            │
                  ┌─────────┼─────────┐
                  │通过      │拒绝      │退回补充/挂起(保持本态)
                  ▼          ▼
                已通过     已拒绝
    （任意非终态可因撤回/过期 ──→ 已关闭）
```

### 2.2 状态说明

| 状态 | 类型 | 说明 |
|---|---|---|
| 待审核 | 非终态 | 申请已接收，尚未进入引擎处理 |
| 审核中 | 非终态 | 引擎正在跑决策流（自动），或人工尚未介入 |
| 待人工复核 | 非终态 | 引擎无把握自动判定，已"转人工"；等待人拍板（可多轮） |
| 已通过 | 终态 | 自动或人工给出"准入"结论 |
| 已拒绝 | 终态 | 自动或人工给出"拒绝"结论 |
| 已关闭 | 终态 | 撤回 / 过期 / 重复申请等，未产生有效结论 |

## 3. 各状态下一步操作矩阵

| 审核状态 | 决策结果 | 可由谁操作 | 下一步操作 |
|---|---|---|---|
| **待审核** | 处理中 | 系统 / 录件员 | 提交进入引擎 → 审核中；或补齐材料后提交 |
| **审核中** | 处理中 | 系统（自动） | 引擎跑决策流，自动出 已通过/已拒绝；人工可干预"转人工" → 待人工复核；可看实时进度 |
| **待人工复核** | 待人工判定 | 复核员 / 主管 | 打开复核工作台 → 通过/拒绝（必填理由）/退回补充材料/挂起/转主管审批；SLA 计时 |
| **已通过** | 准入 | 系统 / 信审 | 进入授信放款前置；自动加入贷中监控；发准入通知；导出报告；限期内可复议 |
| **已拒绝** | 拒绝 | 系统 / 信审 | 发合规拒绝通知（不泄露模型细节）；提供复议入口；导出报告；冷静期后归档 |
| **已关闭** | 关闭 | 系统 / 管理员 | 归档；查看关闭原因（撤回/过期/重复）；不可再操作 |

> 注：只有"审核中"和"待人工复核"才可能出现"转人工"动作；"已通过/已拒绝"之后不再有转人工。

```
┌─ 申贷审核详情 · 申请编号 #2026XXX ───────────────────┐
│ 申请人:张*三  产品:消费分期  渠道:App  额度:50,000    │
│ [决策结果:待人工判定]  [审核状态:待人工复核]  SLA 02:14 │
├────────────────────────────────────────────────────┤
│ [申请信息] [审核结论] [人工复核记录] [操作日志]        │
├────────────────────────────────────────────────────┤
│ Tab1 申请信息: 基本信息/证件材料(OCR人脸)/填写异常高亮 │
│               /设备环境/关联信息(同设备同人历史)        │
│ Tab2 审核结论: 决策流路径(可追溯)/命中规则模型/三项评分 │
```

## 5. 多轮人工复核记录设计

### 5.1 设计要点

人工复核经常不是一次完成——可能"转人工 → 要补材料 → 用户补了 → 电话核实 → 再拍板"，中间还有复核员之间、复核员与主管的内部沟通。**必须用一个"会话式时间线"存储，而不是一个结论字段。**

- **待人工复核 这一个状态可以承载无限多轮沟通**，每一轮都是审计留痕；
- 只有最后一条"提交结论"才把状态推到 已通过/已拒绝；
- 该时间线组件可复用"工单与支持"模块的多轮会话设计。

### 5.2 单条记录结构

| 字段 | 说明 |
|---|---|
| 时间 | 精确到秒 |
| 参与方 | 系统 / 复核员 / 主管 / 申请人（区分对客 vs 内部） |
| 动作类型 | 转人工 / 补充材料请求 / 材料提交 / 电话核实 / 内部备注 / 提交结论 / 主管审批 |
| 内容 | 文字说明 |
| 附件 | 补充材料、通话录音摘要等 |
| 状态变更 | 本次操作前后的审核状态（即使不变也留痕） |

### 5.3 参与方分类（重要）

- **对客沟通**：补材料请求 → 申请人上传 → 确认收到。申请人侧可见（脱敏）。
- **内部协作**：复核员备注、主管审批、转交他人。仅内部可见。

### 5.4 多轮示例

```
14:02  系统      转人工            → 待人工复核
14:10  复核员A   补充材料请求[内部]  "请补收入证明"        待人工复核(不变)
14:35  申请人    材料提交[对客]     上传:收入证明.pdf       待人工复核(不变)
15:20  复核员B   电话核实[内部]      "已电核，单位属实"     待人工复核(不变)
15:40  复核员B   提交结论[内部]      "通过，理由:材料齐全"  → 已通过
```
