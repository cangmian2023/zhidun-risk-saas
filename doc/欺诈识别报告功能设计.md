# 欺诈识别报告 · 功能设计

> 适用产品：同盾风格「贷前风控 SaaS」— 申贷审核主线 · 欺诈识别（反欺诈）模块
> 文档定位：欺诈识别环节的「展示 + 判定 + 操作」载体设计，供信审专员、反欺诈策略岗、风控管理员使用。
> 关联文档：`doc/申贷审核功能详细设计.md`、`doc/信息核验报告功能设计.md`、`doc/信用风控报告功能设计.md`、用户提供的《金融风控贷前审核-欺诈识别完整设计文档》

## 一、功能定位与边界

欺诈识别报告是**申贷审核流程中「欺诈识别（反欺诈）」环节的结果载体**。它在「信息核验」「信用风控」之后（同盾一体化下可并行）执行，把"用户入参 + 设备环境数据 + 内部欺诈库 + 同盾跨行业联防（团伙/设备/黑名单）+ 规则引擎/ML 模型/图计算 → 交叉融合 → 决策分流"的全过程，沉淀为一份可被查看、可被操作、可被监管留痕的结构化报告。

### 1.1 与「信息核验报告」「信用风控报告」的关系
| 维度 | 信息核验报告 | 信用风控报告 | 欺诈识别报告 |
|---|---|---|---|
| 关注点 | 信息**真假**（证件/实名/设备） | 信用**历史、全行业借贷行为** | 骗贷**意图**（伪造/冒用/团伙/撸贷） |
| 数据源 | 公安/运营商/银联/设备/联防 | 人行征信/内部信用库/同盾联防/失信/运营商 | 设备指纹库/规则引擎/ML模型/图计算/材料OCR |
| 输出分 | 核验置信分 | 信用综合分（同盾分） | 欺诈分（智信分） |
| 触发 | 进件提交即触发 | 信息核验完成后触发（可并行） | 进件提交即并行触发 |

### 1.2 报告要回答的 5 个问题
| 问题 | 对应报告区块 |
|---|---|
| 系统**聚合了哪些入参**（含设备环境）？ | 3.1 用户提交与入参（原始） |
| 这些入参被**加工成哪些欺诈特征**？ | 3.2 结构化欺诈特征 |
| 每个引擎**单项**判定如何？ | 3.3 单项欺诈判断结果 |
| 反欺诈引擎**是怎么跑的**？ | 3.4 欺诈识别过程 |
| 最终**欺诈结论**是什么？ | 3.5 欺诈结论（含交叉融合） |

### 1.3 报告上的两类操作
- **单条欺诈异常操作**（3.6）：信审员对**某一条异常**（如"团伙关联 G-014""模拟器命中"）做二次核验、标记、备注、电核。
- **整份报告全局操作**（3.7）：对整份报告做自动通过 / 人工放行 / 退回补件 / 直接拒绝 / 提交复审 / 风险入库 / 存证 / 归档等。

### 1.4 输入 / 输出
- **输入**：进件基础入参 + 设备与环境数据 + 内部欺诈存量库 + 外部联防/规则/模型回执 + 信息核验/信用风控异常结果（交叉输入）。
- **输出**：① 机器结构化报告（JSON，给决策引擎/天策调用）；② 人工可视化报告（本设计主体）；③ 监管存证包（设备指纹回执+关联图谱+操作日志）。

## 二、报告核心数据模型（字段总览）

```
Report
├─ report_id              报告唯一 ID
├─ apply_no               进件申请号
├─ applicant              申请人基础信息（脱敏）
├─ status                 报告状态（待审核/审核中/已通过/已拒绝/已退回/已归档）
├─ raw_inputs             ① 用户提交与入参（基础 + 设备环境 + 内部存量 + 外部请求）
├─ structured            ② 结构化欺诈特征（设备风险/行为异常/团伙关联/材料真伪）
├─ single_results[]       ③ 单项欺诈判断结果（5 大引擎各自结果）
├─ fraud_process          ④ 欺诈识别过程（并行任务、耗时、回执、熔断）
├─ fraud_conclusion       ⑤ 欺诈结论（交叉融合 + 智信分 + 风险等级 + 标签 + 分流）
├─ item_actions[]         ⑥ 单条欺诈异常操作记录
├─ report_actions[]       ⑦ 整份报告全局操作记录
└─ audit_trail            全链路操作日志（监管留痕）
```

> 设计原则：**原始入参、结构化特征、单项结果、识别过程、欺诈结论、单条操作、整体操作** 7 个区块一一对应产品诉求。

## 三、报告功能模块详解

### 3.1 用户提交与入参（原始）
**目的**：还原反欺诈引擎的"输入基准"——满足监管"数据来源可追溯"要求。

展示内容（四类分组）：
- **用户结构化基础入参**：姓名、身份证号、手机号、银行卡、申请金额/期限、年龄、收入、工作单位。
- **设备与环境数据**（欺诈识别独有）：设备指纹ID、设备型号、是否模拟器/多开/Root、IP 类型（真实/代理/翻墙）、GPS、传感器序列、SDK 采集的运行环境。
- **内部自有存量数据**：本行欺诈黑名单、灰名单、历史设备库、历史申请行为序列。
- **外部第三方请求**：设备指纹库比对、同盾联防调用、规则引擎触发、材料 OCR/篡改检测调用。

交互：分组折叠展示；点击某条外部请求可展开调用状态/授权时间/原始回执ID。

### 3.2 结构化欺诈特征
**目的**：展示系统把"原始四源"加工成的可比欺诈特征，作为后续打分的输入。

展示内容（特征表）：
| 特征类 | 字段 | 示例值 |
|---|---|---|
| 设备风险 | 是否模拟器/多开/Root、是否群控、设备复用次数、设备风险分、IP 风险 | **模拟器命中** / 设备复用 5 次 / 设备风险分 92 |
| 行为异常 | 近7/30天申贷平台数、异地/跨境申请、新号(入网<30天)撸贷 | 近30天 12 家 / **异地集中** |
| 团伙关联 | 关联身份证/手机号/设备数、团伙风险等级、中介特征 | 关联设备 5 / **G-014(高)** |
| 材料真伪 | 流水 PS、营业执照伪造、收入证明造假、人证不符 | **PS 流水** |

交互：异常特征高亮（如模拟器命中红标）；可展开"特征→数据源"血缘。

### 3.3 单项欺诈判断结果
**目的**：呈现 5 大反欺诈引擎**各自独立**的判定状态，便于定位"哪一类欺诈风险"。

5 大引擎卡片（每个含：状态徽标 / 输出字段 / 回执编码 / 耗时 / 风险标签）：
1. **设备风险引擎**：模拟器/多开/Root/群控识别、设备指纹黑名单碰撞、设备复用。
2. **规则引擎**：数百条反欺诈规则命中（IF-THEN），如"30 天多头≥10 家→拒"。
3. **ML 欺诈评分（智信分）**：模型输出的欺诈概率分（0–100）。
4. **图计算·团伙关联**：GNN 关系网络，共用设备/手机号/地址的批量骗贷识别。
5. **材料真伪**：OCR + NLP + 图像篡改检测（流水/执照/收入证明）。

状态三态：`通过(绿)` / `存疑(黄)` / `命中(红)`。

### 3.4 欺诈识别过程
**目的**：具象化"并行执行 + 熔断降级"，让信审员/运维看到引擎真实执行轨迹，而非黑盒。

展示内容：
- **并行任务时间线**：1 次 `submit` 拆 5 条独立线程（设备风险 0.3s / 规则引擎 0.05s / ML 模型 0.9s / 图计算 0.8s / 材料 OCR 1.1s），各自下发/返回时间戳、耗时。
- **单渠道调用日志**：每条约 引擎 / 请求参数脱敏 / 返回码 / 原始回执ID / 状态。
- **熔断降级标记**：某渠道超时时显示"XX数据缺失（已熔断，不影响其他引擎）"。
- **聚合节点**：全部渠道返回后进入交叉融合计算的聚合时间戳。

### 3.5 欺诈结论（含交叉融合）
**目的**：报告的"结论页"——把单源结果融合为交叉冲突、欺诈分、标签与分流指令。

三部分：
1. **交叉融合清单 `cross_fusion`**：每条标注严重度（黄/红）与冲突来源。
   - 例🔴：模拟器命中 + 近30天12家多头 → 欺诈设备撸贷（设备风险 VS 同盾联防）
   - 例🔴：单一客户信息正常，但关联5个逾期/欺诈账号(G-014) → 团伙欺诈（图计算 VS 内部/设备）
   - 例🔴：人证不符 + 设备为已知欺诈设备 → 冒用身份申贷（信息核验 VS 设备风险）
2. **欺诈分（智信分）**：0–100，分数越高欺诈风险越高；由设备风险 + 规则命中 + 模型分 + 团伙关联 + 材料异常加权计算。
3. **风险等级 + 风险标签 + 决策分流指令**：
   - 低风险（分≤20）：无异常 → **自动通过**
   - 中风险（分 20–80）：模拟器疑似/异地申请/轻度异常 → **人工复核 / 退回补件**
   - 高风险（分≥80）：模拟器命中/团伙关联/材料造假/冒用身份 → **直接拒绝**

### 3.6 操作人对单条欺诈异常的进一步操作
**目的**：信审员对**某一条异常**（如"团伙关联 G-014"）做人工干预，纠正误报、补充核验或确认欺诈。

单条操作类型（每个 single_result 卡片右上角"操作"菜单）：
| 操作 | 说明 | 落库 |
|---|---|---|
| 调关联图谱 | 查看共用设备/手机号/地址的关系网，可下钻 | item_actions + audit_trail |
| 二次核验 | 重新调用该引擎（如二次设备指纹/材料 OCR） | 同上 |
| 电话电核 | 核实负债/多头真实用途，记录通话笔录 | 同上 |
| 风险标记 | 区分「客户合理行为」/「疑似欺诈」/「团伙欺诈」 | 同上 |
| 补充佐证 | 上传收入流水、结清证明，豁免部分风险 | 同上 |
| 添加备注 | 文字说明（必填理由） | 同上 |

约束：单条操作**不影响**其他源结果，也不自动改变整份报告分流；需结合 3.7 整体操作生效。

### 3.7 整份欺诈识别报告的全局操作
**目的**：对整份报告做出最终业务处置，是欺诈识别流程的出口。

整体操作类型：
| 操作 | 触发条件/适用 | 系统动作 |
|---|---|---|
| 自动通过 | 全绿、分≤20 | 进入授信定价 |
| 人工放行 | 中风险经核实为误报，推翻系统拒贷 | 正常授信，记录理由 |
| 退回补件 | 材料存疑可补（流水/执照待核） | 向用户下发补件通知 |
| 直接拒绝 | 红标/分≥80（模拟器/团伙/材料造假/冒用） | 记录欺诈台账，终止流程 |
| 提交复审 | 高风险大额件 | 推送二级审核双人复核 |
| 风险入库 | 确认欺诈 | 加本地黑名单 + 回流联防样本库 |
| 导出/存证/归档 | 任意/终态前 | PDF 导出；设备指纹+图谱+日志存证；留存≥5年 |

权限：信审专员可"转人工/退回/备注/电核"；反欺诈审核员（二级）可"拒绝/通过/放行"；反欺诈策略岗可"配置规则/名单库/样本回流"。

## 四、欺诈识别列表页设计（涉及的列表页修改）

> 当前 `src/console/specsCr.ts` 的 `cr:pre-fraud`（欺诈识别列表页）已有 6 列（进件号/申请人/设备环境/群控模拟器/异常行为/团伙），但**未挂接报告详情**（无 `reportKey`/`viewNavigate`），是"断头列表"。
> 为支撑本报告载体，建议对列表页做如下改造（**设计建议，待确认后落地代码**）：

### 4.1 新增/调整列
| 列 | 字段 | 说明 |

| 设备环境 | `device` | 保留（徽标） |
| 群控/模拟器 | `simu` | 保留（徽标） |
| 异常行为 | `beh` | 保留（徽标） |

## 五、报告状态机与权限

```
            submit(进件触发, 与信息核验/信用风控并行)
              │
              ▼
         [待审核] ──自动全绿(分≤20)──▶ [已通过] ──▶ [已归档]
              │
              ├─中风险──▶ [审核中] ──信审判定──▶ [已通过]/[已退回]/[已拒绝] ──▶ [已归档]
              │
              └─红标(模拟器/团伙/材料造假/冒用)──▶ [已拒绝] ──▶ [已归档]
```

- 任一终态（通过/拒绝/退回）前，必须完成**存证**。
- 所有状态变更写入 `audit_trail`（操作人、时间、前后值、理由）。

## 七、与同盾方案的映射（实现参考）

| 本报告区块 | 同盾对应能力 |
|---|---|
| 3.1 原始入参 | 机构 `submit` 进件报文（信息核验+信用风控+欺诈识别同接口触发） |
| 3.2 结构化特征 | 同盾指标解析引擎（设备/行为/团伙/材料拆解） |
| 3.3 单项结果 | 单引擎独立返回（设备风险/规则/ML/图计算/材料真伪） |
| 3.4 欺诈识别过程 | 分布式多线程并行引擎（与信息核验、信用风控同步执行） |
| 3.5 交叉融合 | 智信分（欺诈分）+ 交叉冲突标签 |
| 3.6 单条操作 | 信审后台"疑点复核"单源操作 + 云图关联图谱展开 |
| 3.7 整体操作 | 天策决策引擎分流 + 名单管理 + 样本回流 |

> 同盾差异化核心：① 信息核验/信用风控/**欺诈识别同一接口同步执行**（一体化并行）；② **跨行业联防欺诈库**（自建系统无跨机构多头/团伙数据）；③ **团伙欺诈图谱**（云图 GNN 穿透关联）；④ 风险**交叉联动**（证件虚假+模拟器+多头叠加）。本 SaaS 原型按同盾能力完整呈现上述差异。

## 八、验收要点（开发 checklist）
- [ ] 原始入参：四类数据（基础/设备环境/内部/外部）均可展示、可追溯
- [ ] 结构化特征：设备风险/行为异常/团伙关联/材料真伪特征齐全，异常高亮
- [ ] 单项结果：5 引擎状态徽标、回执码、耗时、标签齐全
