# 信息核验 · 列表与报告详情 框架 / 交互 / 分区内容总结

> 依据前端实现代码整理（`src/console/InfoVerifyList.tsx`、`PreVerifyDetail.tsx`、`VerifyOps.tsx`、`infoVerifyReport.ts`、`specsCr.ts`、`Console.tsx`）。
> 信息核验是「贷前审核」主线的一环，定位为**核验 / 反欺诈 / 信用交叉结果 + 工单人工处置**，操作随审核状态动态变化。

---

## 一、整体结构（两个页面 + 一套共用操作引擎）

| 页面 | 文件 | 路由 key | 说明 |
|---|---|---|---|
| 信息核验列表 | `InfoVerifyList.tsx` | `cr:pre-verify` | 进件清单、统计概览、筛选、行内操作 |
| 信息核验报告详情 | `PreVerifyDetail.tsx` | `cr:pre-verify-detail` | 单笔进件的完整核验报告 |
| 状态 / 操作引擎（列表与详情共用） | `VerifyOps.tsx` | — | 操作矩阵、5 个操作弹窗、行/栏渲染 |
| 报告数据构建 | `infoVerifyReport.ts` | — | 三态样例（通过/预警/拒绝）数据 |

**导航关系**：列表 → 点击「申请编号」或「查看」→ 携带 `sys / work / op / id` 四个 URL 参数跳转到详情页（`/console/cr/pre-verify-detail?...`）。详情页据此还原该行的**系统结果 / 工单状态 / 操作人员 / 编号**，保证列表与详情状态一致。返回走 `onBack → /console/cr/pre-verify`。

**两套独立标签体系**（沿用申贷审核语义）：
- 自动审核（`SysResult`）：处理中（灰）/ 通过（绿）/ 拒绝（红）/ 预警（琥珀）→ **DecisionTag 实色胶囊**
- 人工审核（`WorkStatus`）：核验计算中 / 待确认 / 已确认 / 待审核 / 已提交双人复核 / 双人复核-通过 / 双人复核-拒绝 / 强制放行办结 → **StatusTag 带点描边**

## 二、列表页（InfoVerifyList）

### 2.1 顶部 4 个统计项（StatCard）
| 指标 | 口径 | 强调色 |
|---|---|---|
| 待人工处置 | 核验计算中 / 待确认 / 待审核 | 琥珀 |
| 自动审核通过率 | 系统通过数 / 总笔数 | 翠绿 |
| 待双人复核 | 已提交双人复核笔数 | 紫罗兰 |
| 强制放行(高敏感) | 强制放行办结件数 | 玫红 |

### 2.2 筛选条件（Panel 卡片，沿用申贷审核筛选卡外观）
- 文本搜索：申请编号 / 申请人；审核人
- 多选 Chip：产品（信用贷/抵押贷/经营贷）、渠道（APP/H5/小程序/线下）、自动审核、人工审核
- 数值区间：异常值 ≥（综合打分）、申请额度 ≤
- 下拉（SingleSelect，可清空）：申请时间（近 7/30/90 天）
- 「重置」按钮
- 计算口径：`now` 以 `2026-06-20 23:59:59` 为基准做时间过滤；其余为本地 `useState` 内存过滤，分页 50 条/页。

### 2.3 列表表格
- **冻结列**：左冻结「申请编号 / 申请人」两列，右冻结「操作」列；中间横向滚动（`position: sticky`，固定列宽不换行）。
- **列**：申请编号 | 申请人 | 产品 | 渠道 | 申请额度(¥，右对齐) | 综合打分(异常值 0–100，随自动审核上色：绿/琥珀/红/灰) | 自动审核(DecisionTag) | 人工审核(StatusTag) | 审核人 | 申请时间 | 操作。
- **样例数据**：14 行（`seedRows`），严格对齐交互说明——自动审核（通过<20 / 预警20–60 / 拒绝>80 / 处理中低）× 人工审核 × 审核员 × 异常值分档，覆盖全部 8 种工单状态。
- **行内操作**：`VerifyRowActions`（见第三章），查看按钮在「核验计算中」时置灰。

## 三、操作矩阵与弹窗（VerifyOps，列表与详情共用）

### 3.1 操作矩阵 `opsFor(sys, work)`
按「系统自动审核结果 × 工单人工状态」推导可执行按钮：

| 自动审核 | 工单状态 | 可执行按钮 |
|---|---|---|
| — | 核验计算中 | 查看（置灰） |
| 通过 | 待确认 | 查看、**报告确认** |
| 通过 | 已确认 | 查看 |
| 拒绝 | 待确认 | 查看、**报告确认**、**强制复审** |
| 拒绝 | 已确认 / 强制放行办结 | 查看 |
| 预警 | 待审核 | 查看、**提交双人复核** |
| 预警 | 已提交双人复核 | 查看、**确认放行**、**确认拒绝** |
| — | 双人复核-通过 / 双人复核-拒绝 | 查看（已办结） |

### 3.2 五个操作弹窗（均带附件上传区 `AttachmentDrop`，纯前端 mock）
1. **整体报告确认** `ReportConfirmModal`：归档背书，不改结果；提交后工单置「已确认」。
2. **确认放行** `ConfirmPassModal`：需填放行依据 + 必传佐证 + 勾选承诺；→「双人复核-通过」。
3. **确认拒绝** `ConfirmRejectModal`：需填拒贷具体原因；→「双人复核-拒绝」。
4. **提交双人复核** `SubmitDualReviewModal`：填核查说明；提交后工单锁定流转主管，→「已提交双人复核」。
5. **强制复审** `ForceRecheckModal`（高敏感）：红色警示，需填误判理由 + 必传佐证 + 勾选担责；推翻系统拒贷，→「强制放行办结」。

> 所有操作通过 `onApply(Partial<VerifyRow>)` 更新行状态并触发 toast 提示（`useModule().flash`）。

### 3.3 列表行 vs 详情栏
- **列表行** `VerifyRowActions`：纯文字链接按钮（查看/报告确认/强制复审/提交双人复核/确认放行/确认拒绝），不含状态标签（状态已在列展示）。
- **详情栏** `VerifyActionBar`：上方展示「系统结果 / 工单状态 / 操作人员」三标签 + 同套操作按钮，与列表完全一致（详情页默认 `showView=false`）。

## 四、报告详情页（PreVerifyDetail）框架

布局：**左侧主内容区（8 个区块）+ 右侧 sticky 章节导航（7 个锚点）**。

### 4.1 领衔区块（报告顶部）
- **异常值模型卡 `ScoreModelCard`**：综合风险模型领衔整份报告。
  - 总分大数字 `异常值/满分(100)` + 风险等级胶囊（高/中/低）+ **阈值刻度条**（0–20 绿 / 20–80 琥珀 / 80–100 红，指针定位当前分）。
  - **构成项分解表**：风险维度 / 用户情况 / 得分 / 权重 / 等级；核心因子高亮；点击「核心因子」行可跳转定位到对应疑点（scrollIntoView）。
  - 判定规则说明 + 审计栏（计算时间 / 规则版本 / 综合报告 ID）。
- **结论与终审操作卡**：`VerifyActionBar`（系统结果/工单状态/操作人员 + 操作按钮）下方弱化展示「核验过程」横向步骤条（每步按自身结论上色：预警黄/拒绝红/通过绿/待定灰）。

### 4.2 七个内容区块（Panel）+ 章节导航锚点
| 顺序 | 区块标题 | 内容要点 |
|---|---|---|
| 一 | 用户基本信息 `basic` | 11 项基础字段（姓名/身份证/手机/银行卡/开户行/年龄/学历/单位/收入/住址/婚姻），`valid` 异常标「格式异常」红标；下方「环境采集」5 项（设备指纹/IP/GPS/渠道/APP 版本）。 |
| 二 | 用户证件照 `images` | 身份证人像面 / 国徽面 / 活体人脸(视频) / 银行卡，4 张；每张含 OCR 识别结果；图片失败有占位降级。 |
| 三 | 多源并行核验单项报告 `single` | 5 张 `SingleCard`：公安实名 / 银行卡四要素 / 运营商实名 / 终端设备风险库 / 跨行业联防联控。每卡含图标、调用状态(成功/失败/部分)+耗时、结论胶囊、风险结论文案、关键字段行、合规留痕（核验流水/时间/授权/渠道）；右上「操作」下拉。 |
| 四 | （弱化）核验过程 | 见 4.1，置于结论卡底部横向步骤条。9 步：填报→活体→设备指纹→环境→格式→OCR→多源单项→交叉融合→模型打分。 |
| 五 | 数据交叉融合综合报告 · 5 项 `cross` | ① 顶层 5 项评估维度 `AtomicCard`（身份真实性/银行卡归属/设备安全性/入网稳定性/关联风险，正常绿/预警琥珀/高危红，可点跳转关联疑点）；② 综合风险头部栏（最终结论胶囊 + 综合风险等级 + 异常值 + 风险标签 chips 可点定位）；③ 判定规则说明 + 一句话总结 + 审计栏；④ 多源风险交叉疑点明细 `ConflictCard`（核心拒贷因子高亮、可展开/收起、数据源匹配、处理逻辑、本条豁免/备注/上传佐证）。 |
| 六 | 单项核验全量操作日志 `merged-ops` | `MergedOpTable`：合并「单项操作(itemActions) + 用户操作日志(opLogs)」，按时间排序；列：操作对象 / 类型 / 标签 / 操作人 / 时间 / 变更前 / 变更后 / 原因备注&附件 / 复核状态。 |
| 七 | 整体操作 `report-actions` | 整体报告确认动作（reportActions，归档背书）。 |

> 注：设计文档中还有「结构化字段比对 `struct`」（OCR 原文→解析→来源→匹配→差异，如「工商银行→中国工商银行」补全）数据已构建，但当前详情页**未渲染该区块**（可作为后续补充点）。

### 4.3 详情页交互能力
- **右侧 sticky 导航**：7 个锚点（异常值/基本信息/证件照/单项报告/交叉报告/操作日志/整体操作），按各区块是否有异常自动标红/绿点；点击平滑滚动。
- **风险标签联动**：交叉报告头部的风险标签 chips 点击 → 平滑定位到对应疑点卡片（`conflict-<id>`）。
- **异常值构成项联动**：构成表中 core 项点击 → 定位到疑点详情。
- **原子维度卡联动**：有疑点的 `AtomicCard` 点击 → 定位到对应 `ConflictCard`。
- **单项操作下拉 `SingleCard`**：按结论推导——拒绝/预警/待定→重新核验；非通过→录入备注；预警→标记豁免；全部→查看原始回执；预警→关联电核。
- **全局弹窗**：录入备注（带附件）、原始回执（mock JSON）、标记豁免（两步走：申请→二级复核，独立复核人）、查看打分权重明细（各风险项权重占比 + 策略原文 V2.6）。
- **操作日志实时追加**：重新核验 / 录入备注 / 标记豁免 / 查看回执 / 关联电核 均写入本地 `opLogs` 并即时体现在「全量操作日志」表。
- **返回顶部**悬浮按钮（右下角）。

## 五、数据驱动说明（infoVerifyReport）

- `buildInfoVerifyReport(id?)`：按传入 `id` 关键字选套餐——含 `REJECT`→拒绝(红)；含 `MANUAL/WARNING`→预警(黄)；否则→通过(绿)。
- 三态共用同一份基础数据（申请人「张*伟 / 110***********0314」），通过改 `decision / threads / cross.* / single[].conclusion` 呈现不同风险等级：
  - **通过**：异常值 0、低风险、无冲突、全绿。
  - **预警**：异常值 58、中风险、2 条中风险冲突（c2 关联逾期 / c3 新手机号）、设备/入网预警。
  - **拒绝**：异常值 82、高风险、3 条冲突（c1 设备群控=核心拒贷因子 / c2 / c3）、风险标签含「设备群控(红)」。
- 当前为前端 mock 数据，无后端接口；`struct`（结构化字段比对）已建模但未在详情页渲染。

## 六、已知待办 / 优化点（供参考）
1. **`../store` 缺失**：`InfoVerifyList.tsx` 与 `PreVerifyDetail.tsx` 均 `import { useModule } from '../store'`，但 `src/store` 文件不存在 → Vite 构建报错（Failed to resolve import "../store"）。需补 `src/store.ts`（提供 `useModule()`，至少返回 `{ flash }`）。
2. **结构化字段比对区块未渲染**：`struct` 数据已构建，可补一个「结构化字段比对」Panel（OCR 原文→解析→来源→匹配→差异）。
3. **详情页跳转参数驱动**：列表跳详情仅传 `sys/work/op/id` 四参，详情页按 `sys` 映射到套餐（REJECT/WARNING/PASS）；若需真正按具体进件号取不同数据，需在 `buildInfoVerifyReport` 接入进件级数据。
4. **强制放行/双人复核操作仅本地状态**：操作结果只更新前端 `rows`/`verifyRow`，无后端落库与真实双人复核流程。
