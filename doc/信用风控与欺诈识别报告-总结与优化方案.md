> 适用产品：同盾风格「贷前风控 SaaS」— 申贷审核主线
> 文档定位：① 厘清信用风控报告、欺诈识别报告的功能与目的；② 复盘已完成的「申贷审核报告」「信息核验报告」的页面/逻辑/功能/样式/交互；③ 基于两个样板报告的可复用能力，给出信用风控、欺诈识别两份报告的进一步优化方案。
> 关联代码：`src/console/CreditReportDetail.tsx`、`src/console/FraudReportDetail.tsx`、`src/console/PreApplication.tsx`、`src/console/PreVerifyDetail.tsx`

## 0. 背景与目标

- **现状**：信用风控报告（`CreditReportDetail.tsx`）与欺诈识别报告（`FraudReportDetail.tsx`）已搭好骨架，但"功能不准确"——详情页与列表/多状态数据脱节、交互以静态展示为主、完成度落后于已完成的「申贷审核报告」与「信息核验报告」。
- **目标**：先厘清四份报告的功能定位，再把申贷审核 / 信息核验两个"样板"的可复用能力迁移到信用风控 / 欺诈识别，给出可落地的优化方案（本方案聚焦"方案"，代码落地按需排期）。

## 1. 信用风控报告（CreditRiskReport）— 功能与目的

### 1.1 功能定位
- 申贷审核主线中「信用风控（联防联控）」环节的结果载体；在「信息核验」通过后（可并行）执行。
- 把"用户入参 + 内部存量 + 人行征信 + 同盾跨行业联防 + 失信/运营商履约 → 指标计算 → 交叉融合 → 决策分流"的全过程，沉淀为一份可查看、可操作、可监管留痕的结构化报告。

### 1.2 报告要回答的 5 个问题 / 7 个区块
| 问题 | 对应区块 |
|---|---|
| 系统聚合了哪些入参？ | 一、用户提交与入参（原始） |
| 入参被加工成哪些信用特征？ | 二、结构化信用特征（多头/负债/失信/关联） |
| 每个数据源单项判定如何？ | 三、单项信用判断结果（5 大数据源） |
| 风控引擎是怎么跑的？ | 四、信用风控过程（并行任务 + 熔断） |
| 最终信用结论是什么？ | 五、信用结论（交叉融合 + 同盾分 + 分流） |
| 单条异常怎么干预？ | 六、单条信用异常操作 |
| 整份报告怎么决策？ | 七、整体报告全局操作 + 审计轨迹 |

### 1.3 两类操作
- **单条信用异常操作（3.6）**：对某一联防异常（如"失信被执行人""团伙关联 G-014"）做调取明细 / 二次核验 / 电话电核 / 风险标记 / 补充佐证 / 添加备注。
- **整份报告全局操作（3.7）**：自动通过 / 人工放行 / 降额授信 / 直接拒绝 / 退回补件 / 提交复审 / 风险入库 / 导出存证归档。

### 1.4 目的总结
让信审员"看得见"跨机构信用与联防联控结果，对每一条异常可干预、对整份报告可决策，并满足监管"数据来源可追溯 + 操作全程留痕"的要求。

## 2. 欺诈识别报告（FraudReport）— 功能与目的

### 2.1 功能定位
- 申贷审核主线「欺诈识别（反欺诈）」环节的结果载体；进件提交即与信息核验 / 信用风控并行触发。
- 把"用户入参 + 设备环境数据 + 内部欺诈库 + 同盾联防（团伙/设备/黑名单）+ 规则引擎/ML 模型/图计算 → 交叉融合 → 决策分流"的全过程，沉淀为结构化报告。

### 2.2 报告要回答的 5 个问题 / 7 个区块
| 问题 | 对应区块 |
|---|---|
| 系统聚合了哪些入参（含设备环境）？ | 一、用户提交与入参（原始） |
| 入参被加工成哪些欺诈特征？ | 二、结构化欺诈特征（设备/行为/团伙/材料） |
| 每个引擎单项判定如何？ | 三、单项欺诈判断结果（5 大引擎） |
| 反欺诈引擎是怎么跑的？ | 四、欺诈识别过程（并行 + 熔断） |
| 最终欺诈结论是什么？ | 五、欺诈结论（交叉融合 + 智信分 + 分流） |
| 单条异常怎么干预？ | 六、单条欺诈异常操作 |
| 整份报告怎么决策？ | 七、终审操作 + 审计轨迹 |

### 2.3 两类操作（结构同信用报告，差异在引擎与标签语义）
- **单条欺诈异常操作（3.6）**：调关联图谱 / 二次核验 / 电话电核 / 风险标记 / 补充佐证 / 添加备注。
- **整份报告全局操作（3.7）**：自动通过 / 人工放行 / 退回补件 / 直接拒绝 / 提交复审 / 风险入库 / 导出存证归档。

### 2.4 目的总结
识别骗贷意图（伪造 / 冒用 / 团伙 / 撸贷），呈现设备风险与团伙图谱，对异常可干预、对整份可决策、监管留痕。

## 3. 参考样板 A：申贷审核报告（PreApplication.tsx + scoreReport.ts + preApp.ts）

### 3.1 页面结构
- 左：进件列表（筛选 / 批量导入 / 历史检索 / 批量转人工）。
- 右：Drawer 详情 —— 申请人画像、设备与环境、风险评分（三张分数卡）、命中规则与模型、决策建议、决策流时间线、人工复核会话。

### 3.2 逻辑（数据驱动、可计算）
- 多样例 A/B/C：`selectSample(r)`（`scoreReport.ts:384`）按 `fraudScore` / `credit` 选样。
- `buildScores()` 生成 智察分 / 智信分 / 智融分 三张分数卡；`buildRules()`（`scoreReport.ts:546`）标记命中 / 未命中 / 权重；`computeRiskIndex()`（`scoreReport.ts:413`）风险指数公式 `0.5*fraud + 0.5*rulePoints`。
- 决策 / 费率由状态推导，并按产品生成差异化 `ProductReport`。

### 3.3 功能
- 三张分数卡 + 阈值刻度进度条（`ScoreBar`，`FRAUD_MARKS` 30/60、`CREDIT_MARKS` 580/720）；命中规则可展开因子 / 组件；按产品生成差异化报告；详情按审核状态动态展示操作（通过 / 拒绝 / 挂起 / 转人工 / 查看进度 / 导出 / 复议 / 关闭）。

### 3.4 样式
- 分数可视化用带刻度进度条；`rose/amber/emerald` 三级预警；证件 / 活体内联 SVG 占位预览。

### 3.5 交互
- 按状态动态操作栏；复议弹窗（合规留痕写入审计日志）；决策引擎运行进度弹窗；报告 HTML 导出；列表批量转人工。

### 3.6 可复用能力清单（供优化复用）
- **A1** 多状态样例 + 按选中进件路由到对应报告数据
- **A2** 分数卡 + 阈值刻度进度条（`ScoreBar` 思路）
- **A3** 动态操作栏 + 弹窗（复议 / 进度 / 导出）
- **A4** 命中规则可展开（因子下钻）

## 5. 当前实现问题诊断（基于代码核对）

### 5.1 信用风控报告 `CreditReportDetail.tsx`
- **[P1] 返回路由错误**：`onBack={() => nav('/console/cr:pre-credit')}`（行 20）→ 应为 `/console/cr/pre-credit`。点击"信用风控列表"会跳到错误路由。
- **[P2] 风险标签被强制黄色**：`kind="amber"`（行 59、154）→ 不论语义一律黄色，丢失"失信 / 团伙 / 重度多头"的严重度区分。
- **[P3] 结论区重复**：顶部结论横幅（行 48–66）与"五、核验结果"区（行 138–164）高度重复，同一决策 / 信用分 / 标签渲染两遍。
- **[P4] 无右侧 sticky 导航**：长报告不易跳转（欺诈报告已有，见 4.1 / 4.5-B1）。
- **[P5] 操作是静态表格**：六 / 七区仅为表格展示，无实际触发（对照 B4 / B5）——没有"调取明细 / 二次核验 / 电核 / 备注"等弹窗，操作不入 `audit_trail`。
- **[P6] 复核 / 提交终审无逻辑**：行 42–43 按钮不驱动状态机，与 B5 / `FinalOpsCard` 缺失形成对照。
- **[P7] 信用分无刻度**：行 56 仅纯数字，缺阈值刻度（对照 A2，应展示 ≤20 低 / 20–80 中 / ≥80 高 的风险刻度）。
- **[P8] 取数依赖正确传参**：`getCreditReport(applyNo)`（`creditReport.ts:93`）按 `apply_no` 取，JSON 含 8 份；但列表 `specsCr.ts:137` 的 `reportKey:'apply_no'` 须与详情取数键一致，否则回退首条——建议两键都支持并加"未匹配"提示。

### 5.2 欺诈识别报告 `FraudReportDetail.tsx`
- **[Q1] 核心 bug — 忽略选中进件**：`const d = buildFraudReport()`（行 22）**完全忽略 `reportId`**（行 21 取到却不用）→ 无论点哪个进件，永远展示 `FR20260721001`"全通过"样例。这是"功能不准确"的最直接原因。
- **[Q2] 仅 1 份写死样例**：`fraudReport.ts` 的 `buildFraudReport()`（行 109–189）只生成"全通过 / 低风险"，而列表展示 fraudScore 8/58/95/90… 多种状态——详情与列表严重脱节，无法演示 命中 / 存疑 / 拒绝。
- **[Q3] 异常判定靠字符串嗅探**：行 99 `includes('命中'/'疑似'/'代理'/'伪造')`、行 117/126 `includes('是'/'疑似'/'高'/'＞')` → 脆弱、非真实规则引擎，改文案即错。
- **[Q4] 三态口径不统一**：单项用 通过 / 存疑 / 命中，结论用 自动通过 / 人工复核 / 直接拒绝 / 退回补件，两套未统一映射。
- **[Q5] FinalOps 丢中间态**：行 243 `isReject ? 'reject' : 'pass'` → 人工复核 / 退回补件 被当 `pass`（与信息核验同一缺陷）。
- **[Q6] 欺诈分无刻度**：行 64 / 197 纯数字，缺阈值刻度（应 ≤20 低 / 20–80 中 / ≥80 高）。
- **[Q7] 缺深度可视化**：设备环境 / 团伙图谱 / 材料真伪仅文本列表，未达"欺诈识别"差异化呈现（对照 B2 / B3）。
- **[Q8] 既有优点（保留）**：返回路由正确（`/console/cr/pre-fraud`，行 45）；已有 sticky 导航（行 270–288）；结论横幅配色合理（行 55–67）——这些优于信用报告，应保留。

> **共性根因**：详情页"写死单一样例 + 忽略列表选中 id"导致与列表 / 多状态数据脱节；操作是"展示"而非"可触发"；分数缺刻度；对两个样板（A/B）的交互能力复用不足。

## 7. 信用风控报告 优化方案（逐项）

### 7.1 数据正确路由（C1，必做）
- `creditReport.ts` 增加 `getCreditReportById(reportId)`；详情行 18 改用 reportId。列表 `specsCr.ts:137` 的 `reportKey:'apply_no'` 须与详情取数键一致，建议两键都支持：`find by apply_no ?? report_id`。
- 防御：取不到时回退首条，并在 UI 顶部提示"未匹配到该进件报告，展示样例"。

### 7.2 返回路由修复（C5，P1）
- 行 20 改为 `nav('/console/cr/pre-credit')`。

### 7.3 风险标签按严重度着色（C3，P2）
- 行 59 / 154 用 `severityKind(tag)` 或新增 `riskTagKind()` 映射：失信 / 团伙 / 重度多头 → red；轻度多头 / 负债偏高 → amber；正常 → green，不再强制 amber。

### 7.4 结论区去重（C6，P3）
- 删除行 48–66 的"结论横幅"，或删除行 138–164 与横幅重复的决策 / 分 / 标签，保留一处。建议：顶部保留精简横幅，五区只放交叉比对明细，决策 / 分不再重复渲染。

### 7.5 加 sticky 章节导航（B1，P4）
- 复制欺诈报告的 `navCards` 模式到信用报告（7 个锚点：raw / struct / single / process / cross / item-actions / report-actions），右侧 `fixed w-44`，alert 红点按"有失败源 / 有交叉冲突 / 有操作"点亮。

### 7.6 信用分阈值刻度条（A2，P7）
- 复用 `scoreReport` 的 `ScoreBar` 思路：行 56 纯数字改为 0–100 刻度条，标注 ≤20 低(绿) / 20–80 中(黄) / ≥80 高(红) 三段刻度，当前值指针落点。

### 7.7 单项操作可触发（B4，P5）
- 每个 `single_results` 卡片右上角加"操作"菜单（调取完整明细 / 二次核验 / 电话电核 / 风险标记 / 补充佐证 / 添加备注），点击打开 Modal（参照 `PreVerifyDetail` 的备注 / 豁免 Modal），结果追加到 `r.item_actions` 并写 `r.audit_trail`（本地 state）。

### 7.8 整体操作接状态机（B5，P6）
- 引入 `FinalOpsCard`（欺诈已用，信用缺失），把"复核 / 提交终审"接成真实动作：按当前 `decision` 与权限展示可用操作（自动通过 / 人工放行 / 降额 / 拒绝 / 退回 / 复审 / 风险入库），点击驱动 `status` 状态机并写 `audit_trail`。

### 7.9 交叉融合疑点点击定位（B3）
- 五区交叉冲突卡片加 `onClick` 滚动到对应 `single_result` 源（参照信息核验"风险标签点击定位"）。

## 8. 欺诈识别报告 优化方案（逐项）

### 8.2 数据驱动异常判定（C3 / 去字符串嗅探，Q3）
- 在 `SingleFraudResult` / `structured` 增加显式字段：`is_abnormal: boolean`、`severity: '低'|'中'|'高'`；高亮与边框由该字段决定，删除行 99 / 117 / 126 的 `includes` 嗅探。
