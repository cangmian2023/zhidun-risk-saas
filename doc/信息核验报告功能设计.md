# 信息核验报告 · 功能设计

> 适用产品：同盾风格「贷前风控 SaaS」— 申贷审核主线 · 信息核验模块
> 文档定位：信息核验结果的「展示 + 判定 + 操作」载体设计，供信审专员、风控策略岗、风控管理员使用。
> 关联文档：`doc/申贷审核功能详细设计.md`、用户提供的《金融风控贷前审核：信息核验完整解析》

---

## 一、功能定位与边界

信息核验报告是**申贷审核流程中「信息核验」环节的唯一结果载体**。它把"用户提交 → 多源并行核验 → 交叉比对 → 决策分流"的全过程，沉淀为一份可被查看、可被操作、可被监管留痕的结构化报告。

### 1.1 报告要回答的 5 个问题
| 问题 | 对应报告区块 |
|---|---|
| 用户**原样**提交了什么？ | 3.1 用户提交内容（原始） |
| 系统把原始内容**结构化**成了什么？ | 3.2 结构化后的内容 |
| 每个数据源**单项**判定结果如何？ | 3.3 单项判断结果 |
| 核验引擎**是怎么跑的**？ | 3.4 核验过程 |
| 最终**综合结论**是什么？ | 3.5 核验结果（含交叉比对） |

### 1.2 报告上的两类操作
- **单项结果操作**（3.6）：信审员对某一个数据源结果（如"运营商实名非本人"）做二次核验、标记、备注、申诉。
- **整体报告操作**（3.7）：对整份报告做通过 / 转人工 / 拒绝 / 退回补件 / 存证 / 归档等。

### 1.3 输入 / 输出
- **输入**：进件提交报文（用户填报 + 影像 + 设备/环境采集）+ 外部数据源回执 + 交叉比对引擎输出。
- **输出**：① 机器结构化报告（JSON，给决策引擎/天策调用）；② 人工可视化报告（本设计主体）；③ 监管存证包（影像+回执+操作日志）。

## 二、报告核心数据模型（字段总览）

一份报告由以下顶层对象组成（完整 JSON Schema 见 `信息核验报告-示例数据.json`）：

```
Report
├─ report_id              报告唯一 ID
├─ apply_no               进件申请号
├─ applicant              申请人基础信息（结构化后）
├─ raw_submission         ① 用户提交原始内容（原文+原图）
├─ structured            ② 结构化后的内容（OCR/解析/标准化）
├─ single_results[]       ③ 单项判断结果（5 大数据源各自结果）
├─ verify_process         ④ 核验过程（并行任务、耗时、回执、熔断）
├─ cross_check            ⑤ 核验结果-交叉比对（冲突列表+综合分+标签+分流）
├─ item_actions[]         ⑥ 单项结果操作记录
├─ report_actions[]       ⑦ 整体报告操作记录
├─ status                 报告状态（待审核/审核中/已通过/已拒绝/已退回/已归档）
└─ audit_trail            全链路操作日志（监管留痕）
```

> 设计原则：**原始内容、结构化内容、判断结果、核验过程、核验结果、单项操作、整体操作** 7 个区块一一对应产品诉求，缺一不可。

## 三、报告功能模块详解

### 3.1 用户提交内容（原始）
**目的**：还原用户进件时的"第一手"输入，作为后续一切核验的比对基准，同时满足监管"原始材料留存"要求。

展示内容：
- **结构化原文**：姓名、身份证号、手机号、银行卡号、开户行、居住地址、工作单位、月收入等填报字段的**原值**（非系统加工）。
- **非结构化原图**：身份证正/反面、人脸活体视频帧、工资流水截图、营业执照等影像，支持放大、下载、水印展示。
- **环境采集原文**：设备指纹、IP、GPS、终端系统、申请时间（用户无感采集，但在报告中如实呈现）。

交互：点击任意影像可查看高清原图；"查看采集元数据"展开设备/IP 原始值。

### 3.2 结构化后的内容
**目的**：展示系统如何把"原始影像+原文"解析为机器可校验的字段，并建立 **原始值 ↔ 结构化值** 的映射，便于信审员发现 OCR 误差或故意篡改。

展示内容（对比表）：
| 字段 | 原始提交值 | 结构化值（OCR/解析） | 差异标记 |
|---|---|---|---|
| 姓名 | 张三 | 张三 | — |
| 身份证号 | 110101199003071234 | 110101199003071234 | — |
| 身份证有效期 | （未填） | 2015.01–2025.01 | ⚠ 已过期 |
| 工作单位 | 北京某某科技有限公司 | 上海某某贸易有限公司（流水OCR） | ⚠ 不一致 |
| 人脸 | 活体视频 | 活体评分 32 / 翻拍攻击疑似 | ⚠ 异常 |

交互：差异行高亮；点击可联动跳转至对应原图/原始填报。

### 3.3 单项判断结果
**目的**：呈现 5 大数据源**各自独立**的核验状态，不掺交叉逻辑，便于定位"哪一类信息有问题"。

5 大数据源卡片（每个含：状态徽标 / 输出字段 / 回执编码 / 耗时 / 风险标签）：
1. **公安实名核验**：状态、人证结论、证件有效性、回执流水号。
2. **运营商实名核验**：三要素匹配、入网时长、是否虚拟号、归属地。
3. **银联银行卡四要素**：四要素匹配、卡类型、发卡行。
4. **设备库核验**：设备风险分、模拟器/多开/ROOT/翻墙标签。
5. **跨行业联防库**：近30天申请机构数、历史骗贷/黑名单、关联图谱。

状态三态：`通过(绿)` / `存疑(黄)` / `失败(红)`。

### 3.4 核验过程
**目的**：具象化"并行执行 + 熔断降级"，让信审员/运维看到核验引擎的真实执行轨迹，而非黑盒。

展示内容：
- **并行任务时间线**：1 次 `submit` 拆 5 条独立线程，各自下发时间戳、返回时间戳、耗时（公安1.5s / 运营商2s / 银联1s / 设备0.3s / 联防0.8s）。
- **单渠道调用日志**：每条记录 数据源 / 请求参数脱敏 / 返回码 / 原始回执ID / 状态。
- **熔断降级标记**：某渠道超时时显示"运营商数据缺失（已熔断，不影响其他源）"。
- **聚合节点**：全部渠道返回后进入交叉比对计算的聚合时间戳。

### 3.5 核验结果（含交叉比对与异常值模型）

**目的**：这是报告的"结论页"——把单源结果融合为冲突清单、异常值（模型化）、风险标签与分流指令。

#### 3.5.1 交叉比对冲突清单 `cross_contradict_list`
每条标注冲突来源（如"公安 VS 运营商"）、冲突描述、严重度（黄/红）。
- 例：公安身份匹配，但运营商手机号实名非本人 → 红。

#### 3.5.2 异常值模型（信息核验综合风险模型 · V2.6）
把"为什么是这个分"结构化为**可复加、可追溯**的模型输出，而非一个黑盒数字。

**① 模型定位**
- 模型名：信息核验综合风险模型；规则版本 **V2.6**（与报告审计栏 `rule_version` 一致）
- 取值域：0–100；**方向"越大越危险"**
- 计算口径：`异常值 = Σ(各风险项得分)`（直接累加，非加权归一）

**② 数据模型 `score_components[]`**（每个构成项一条）
| 字段 | 含义 |
|---|---|
| name | 维度名（如"设备群控"） |
| situation | 该维度下用户实际情况（一句话） |
| score | 该维度得分（正=越危险，累加进异常值） |
| weight | 权重（仅用于展示占比，非归一系数） |
| direction | 风险方向（越大越危险 / 越小越危险） |
| level | 该维度等级（低 / 中 / 高） |
| traceTo | 关联 `atomic` 原子卡 key 或 `conflict` 疑点 id（可追溯联动） |

**③ 阈值刻度与分流**
| 区间 | 等级 | 颜色 | 分流指令 |
|---|---|---|---|
| 0–20 | 低 | 绿 | 自动放行 |
| 20–80 | 中 | 黄 | 转人工复核 |
| ≥80 | 高 | 红 | 直接拒贷 |

**④ 三套样例分解（数据驱动，分项之和 = 总分）**
- **拒绝 82** = 设备群控 60 + 新手机号 14 + 关联逾期 8
- **通过 0** = 无风险证据累计（全部单项核验通过）
- **人工 58** = 设备 30 + 新手机号 18 + 关联 10

> 说明：上述三套为数据层 `CrossCheck.scoreComponents` 的样例口径；与 §5.5「全异常样例」的 94 分属不同场景（全 5 源异常 vs 单 reject 场景），并存不冲突。

**⑤ 前端展示（模型卡）5 块**
1. **模型标识 + 总分卡**：模型名/版本、总分大数字、**方向"越大越危险"**、等级、阈值刻度条（绿/黄/红三段 + 指针落点）
2. **构成项分解表**：`维度 ｜ 用户情况 ｜ 得分 ｜ 权重 ｜ 方向 ｜ 等级`，含**合计行**；核心拒贷因子高亮
3. **可追溯联动**：每个构成项点击 → 定位对应 `atomic` 原子卡 / `conflict` 疑点详情
4. **判定规则与分流**：R1/R2/R3 原文 + 分流指令（≥80 自动拒绝）
5. **合规审计栏**：计算时间 / 规则版本 / 综合报告 ID

**⑥ 判定规则（R1 / R2 / R3）**
- **R1 设备群控命中**（模拟器 / 多开 / ROOT / 翻墙）→ 核心拒贷因子，单项可拉满至 60
- **R2 新手机号**（入网 < 30 天）→ 加 14–18
- **R3 关联逾期**（跨行业联防联控命中）→ 加 8–10

#### 3.5.3 风险标签
由冲突清单与构成项推导得出（如"证件过期""手机号实名非本人""模拟器""多头申贷"），点击标签可定位到对应疑点明细。

### 3.6 操作人对单项结果的进一步操作
**目的**：信审员对**某一个数据源结果**做人工干预，纠正误报、补充核验或确认欺诈。

单项操作类型（每个 single_result 卡片右上角"操作"菜单）：
| 操作 | 说明 | 落库 |
|---|---|---|
| 发起二次核验 | 重新调用该数据源（如二次运营商核验） | item_actions + audit_trail |
| 标记为误报 | 信审判定系统误判（如 OCR 误差） | 同上 |
| 标记为属实/欺诈 | 确认该异常真实存在 | 同上 |
| 补充外部复核 | 手动调其他数据源交叉 | 同上 |
| 添加备注 | 文字说明（必填理由） | 同上 |
| 加入黑白名单 | 该字段/设备加入名单（联动 3.7） | 同上 |

约束：单项操作**不影响**其他源结果，也不自动改变整份报告分流；需结合 3.7 整体操作生效。

### 3.7 整体报告的进一步操作
**目的**：对整份报告做出最终业务处置，是核验流程的出口。

整体操作类型：
| 操作 | 触发条件/适用 | 系统动作 |
|---|---|---|
| 自动通过 | 全绿、分≤20 | 进入信用评分环节 |
| 转人工复审 | 黄标/存疑 | 创建工单，分配信审池 |
| 直接拒绝 | 红标/分≥80 | 记录欺诈台账，终止流程 |
| 退回补件 | 材料缺失可补 | 向用户下发补件通知 |
| 导出报告 | 任意 | 生成 PDF（含影像+回执+操作日志） |
| 存证上链 | 终态前 | 影像+回执+日志打包存证 |
| 加入黑名单 | 确认欺诈 | 身份证/手机号/设备入黑名单 |
| 归档 | 终态后 | 存入信贷档案，留存≥5年 |

权限：信审专员可"转人工/退回/备注"；风控审核员（二级）可"拒绝/通过"；风控策略岗可"配置阈值/黑白名单"。

## 四、报告状态机与权限

```
            submit
              │
              ▼
         [待审核] ──自动全绿──▶ [已通过] ──▶ [已归档]
              │
              ├─黄标──▶ [审核中] ──信审判定──▶ [已通过]/[已退回]/[已拒绝] ──▶ [已归档]
              │
              └─红标──▶ [已拒绝] ──▶ [已归档]
```

- 任一终态（通过/拒绝/退回）前，必须完成**存证**。
- 所有状态变更写入 `audit_trail`（操作人、时间、前后值、理由）。

### 5.5 核验结果（交叉比对 + 综合）
- **交叉冲突清单**：
  1. 🔴 公安身份=张三，但运营商实名=李四（公安 VS 运营商）
  2. 🔴 公安/银联姓名=张三，但银行卡开户人=王五（公安/银联 VS 银行）
  3. 🔴 正常身份 + 设备为模拟器 + 联防多头12次（设备库 VS 联防库）
  4. 🔴 手机号入网7天 + 近15天8家机构申请（运营商 VS 联防库）
  5. 🟡 户籍四川 / 手机号归属河南 / 申请IP境外（属地交叉）
- **综合风险分**：**94**（高危）
- **风险标签**：证件过期、手机号实名非本人、银行卡户主不符、模拟器、多头申贷、团伙关联
- **决策分流**：🔴 **直接拒绝**（红标，终止流程，记录欺诈台账）

## 六、与同盾方案的映射（实现参考）

| 本报告区块 | 同盾对应能力 |
|---|---|
| 3.1 原始内容 | 机构 `submit` 进件报文 |
| 3.2 结构化 | 同盾 OCR/NLP 解析 + SDK 采集 |
| 3.3 单项结果 | 单源 `data_source_list` 独立返回 |
| 3.4 核验过程 | 分布式异步任务调度引擎（并行+熔断） |
| 3.5 交叉比对 | `cross_contradict_list` + 同盾分 |
| 3.6 单项操作 | 信审后台"疑点复核"单源操作 |
| 3.7 整体操作 | 天策决策引擎分流 + 运营监控大盘 |

> 自建系统对比：一般仅能"公安+运营商+银联"三源交叉，**缺设备全域库与跨行业联防库**，故 3.3 中"设备库/联防库"两项异常在自建系统下通常不可见——本 SaaS 原型按同盾能力完整呈现。
